The Solution
Live Now

From fragmented inboxes to one local stream.

chaind syncs supported chats into a normalized local store, then exposes that store through a machine-first IPC interface for scripts, agents, and human operators.

  • Local-first: no public webhook surface required for the default deployment.
  • Machine-first: one schema instead of one parser per platform.
  • Operator-first: communities get one local control plane instead of repeated manual coordination.

Architecture Overview
Live Now

A 5-layer local stack.

5. IPC layerunix socket first - token-aware request handling
4. Store writerSQLite WAL - batched single-writer path
3. Canonical normalizerone message schema - ULID-based identity
2. Protocol adaptersMatrix - Telegram - WhatsApp
1. Daemon coreisolated supervisors - local lifecycle

This is the backbone for the rest of the talk: adapters bring in events, the daemon normalizes them, one writer persists them, and IPC exposes them to automation.

Layer 1 - Reliability
Live Now

Failure stays contained.

Instead of one shared failure path, adapters run under isolated restart loops with capped exponential backoff.

  • One adapter can fail without immediately tearing down the rest of the daemon.
  • Backoff is localized to the failing platform.
  • The core process remains up while the adapter recovers.

Reliability comes from isolation, not luck.

The supervisor loop in the repo is intentionally small: enough structure to restart adapters, not enough magic to hide failure behavior.

Layer 3 - Normalization
Live Now

Agents need one message language.

A local agent should not care whether the input started life as a Matrix event or a Telegram update. It should receive one predictable message shape.

Raw world

Telegram: updates, peers, message IDs

chaind world

One schema: room, author, content, timestamps, reply links

One identity model: ULID-based local IDs

ULIDs keep messages time-sortable and make the local store friendlier for pagination and history traversal.

Layer 4 - Storage
Live Now

A sovereign store with a single writer.

// WAL for concurrent reads
PRAGMA journal_mode=WAL;

// dedicated writer path
SetMaxOpenConns(1);
buffered channel - batch flush

goal: predictable local sync behavior

Messages land in a local SQLite database with WAL enabled. The daemon separates a read pool from a dedicated write path and batches inserts through a store writer.

  • One writer path lowers contention on the local store.
  • FTS5 sits on top for local search.
  • The architecture favors correctness and predictable local behavior over hype benchmarks.
reads pooled writes serialized search FTS5
Layer 5 - IPC
Live Now

The local nervous system.

Default path: a 0600 unix socket under ~/.config/chaind/chaind.sock. That keeps the primary interface local to the machine running the daemon.

Machine-first envelope: JSON over a local IPC surface means Go, Python, Node, shell tools, and local agents can all talk to the same daemon.

There is also an optional local HTTP mirror for some deployments, but the default framing is still local-first IPC.

$ echo '{"cmd":"read"}' | nc -U ~/.config/chaind/chaind.sock

[recv] recent normalized messages...
json in - json out
Security - Permissions
Prototype

Room-scoped permission model.

Tier 0 - Ownerfull access for the operator path
Tier 2 - ReadWriteread and send inside allowed rooms
Tier 4 - SendOnlyannouncements without history access

The middleware is designed around token metadata, room allowlists, and tiered access. That architectural direction is already visible in the daemon.

What is still maturing is the end-to-end provisioning story from CLI issuance to polished operator workflow.

So this slide is deliberately framed as a model and current direction, not as a finished product surface.
Security - Privacy
Prototype

Best-effort local redaction.

chaind includes local redaction hooks for common patterns such as phone numbers, email addresses, and PAN-like strings before data reaches a downstream consumer.

  • Useful for reducing accidental PII exposure in local agent pipelines.
  • Helpful as a guardrail, not as a hard security boundary.
  • Hard guarantees still belong to operator policy and local deployment choices.
{
  "text": "Call me at [REDACTED_PHONE]",
  "meta": {
    "pii_detected": "best_effort"
  }
}

The right claim here is practical local redaction, not magical privacy.
