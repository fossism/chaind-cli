Layer 1: Reliability

Isolated Supervisors.

Most Go daemons use a single errgroup. If one adapter fails, the whole process dies.

chaind uses isolated supervisor loops with exponential backoff. If Telegram MTProto hits a FLOOD_WAIT, it waits alone. Matrix and WhatsApp keep firing.

Reliability through isolation.

Nerd Snipe: Supervisor loop is only 15 lines of Go. Zero dependencies.
Layer 2: Native Protocol

Account Adapters

We bypass Bot APIs entirely. We use the protocols humans use.

  • Telegram: Full MTProto via gotd/td. We handle DC migrations natively.
  • WhatsApp: Reverse-engineered whatsmeow. We act as a linked device.
  • Matrix: Production-grade mautrix-go.

Why MTProto?

Bot API can't read user DMs. MTProto can. chaind gives your agent the same access you have on your phone.

Layer 4: Storage

The Sovereign Store

// SQLite WAL Mode
PRAGMA journal_mode=WAL;
// One writer, many readers
SetMaxOpenConns(1);

GOAL: 100% data integrity under load.

Alice's messages live in a local SQLite DB powered by modernc.org/sqlite (Pure Go).

The Solution: A single StoreWriter Goroutine handles all writes via a buffered channel to prevent SQLITE_BUSY errors.

Zero lag. Zero database locks.

Layer 5: IPC

The Nervous System

Unix Socket (0600): The sovereign choice. No network ingress. Only processes on Alice's machine can talk to chaind.

JSON Envelope: Any language can talk to it. Python, Node, Go, or a simple socat command.

$ echo '{"cmd":"read"}' | nc -U /tmp/chaind.sock

[RECV] 50 messages...
Security: Tokens

Granular Permissions.

Tier 0: Owner Full access (CLI)
Tier 2: ReadWrite Send & Read
Tier 4: SendOnly Announcements

We don't give agents the keys to the castle. We issue Capability Tokens scoped to specific rooms and tiers.

Alice's helpdesk agent can only read the #helpdesk room. It can't see her DMs.
